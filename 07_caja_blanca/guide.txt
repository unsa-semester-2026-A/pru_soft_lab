I. MARCO CONCEPTUAL
I. Introducción
Las técnicas de prueba de caja blanca (basadas en la estructura) consisten en analizar el código y la estructura
del producto a probar y utilizar ese conocimiento para realizar las pruebas. Estas técnicas se emplean en la fase
de pruebas unitarias, aunque también puede aplicarse en otras etapas, como las pruebas de integración. Para su
ejecución, el evaluador o la persona que utilice estas técnicas debe tener un conocimiento profundo de la
tecnología empleada en el desarrollo del programa.

Figura 1 – Técnicas de Caja Blanca. Fuente: instatus
El estándar ISO/IEC/IEEE 29119-4 es un estándar internacional fundamental para pruebas de software, ya que
unifica y estandariza las técnicas de diseño de pruebas. En este estándar se definen técnicas (basadas en la
estructura o de "caja blanca") se centran en derivar casos de prueba a partir del código fuente o la estructura
interna de un programa.

Busca diseñar casos de prueba de manera que cada instrucción ejecutable del código fuente se ejecute al menos

II. Pruebas de Ramas (Branch Testing)
Asegura que cada arista (branch) del Grafo de Flujo de Control (CFG) sea transitada. Esto implica que todas las
ramificaciones posibles de las estructuras de control (if, while, switch) se ejecuten.

III. Pruebas de combinación de condiciones (Branch condition combination testing)
También conocida como Multiple Condition Coverage. Requiere que se prueben todas las combinaciones posibles
de Verdadero y Falso de las condiciones atómicas dentro de una decisión. Si hay $N$ condiciones, se requieren
2N casos de prueba.

IV. Complejidad Ciclomática
La Complejidad Ciclomática es una métrica de ingeniería de software (desarrollada por Thomas McCabe en 1976)
que mide la cantidad de caminos linealmente independientes a través del código fuente de un programa.
A nivel académico, no mide el total de caminos que existen, sino la cantidad de "caminos base". Cualquier otro
camino posible en el código se puede formar combinando estos caminos base. Sirve directamente para definir el
límite superior del número de casos de prueba necesarios para lograr una cobertura completa de ramas (Branch
Coverage).
Se calcula mediante la fórmula del grafo de flujo de control:
CC = A – N + 2
CC = P + 1
Donde: A = aristas, N = nodos de grafo de flujo, P = número de nodos predicado (nodos que contienen una
condición)

Como desarrollador enfocado en la calidad del código y las pruebas, medir la complejidad ciclomática es
fundamental. Mantener este número bajo no solo hace que tu código sea más legible, sino que reduce
drásticamente la cantidad de pruebas unitarias que necesitas escribir para cubrir todos los caminos posibles.
En el ecosistema de Python, usaremos la herramienta Radon para medir la complejidad lógica de un programa.
