1. Introducción

Este informe documenta el proceso completo de validación del comportamiento de un Cajero Automático (ATM) aplicando las metodologías BDD (Behavior-Driven Development) y TDD (Test-Driven Development), siguiendo el flujo de trabajo obligatorio de cinco fases.

El caso de prueba central consiste en verificar que el sistema bloquee correctamente un retiro cuando el monto solicitado supera el saldo disponible, mostrando el mensaje "Fondos Insuficientes". Según Glen Myers, este tipo de prueba de error es fundamental para garantizar la robustez del software.

 

2. Fase 1 – BDD: Definición del Comportamiento (Gherkin)

Antes de escribir cualquier línea de código, se definieron los escenarios en lenguaje natural usando el formato Gherkin (Dado / Cuando / Entonces). Este archivo describe qué debe hacer el programa, no cómo.

5. Fase 4 – TDD Verde: Código Mínimo

Se implementó el código mínimo necesario para que todas las pruebas pasen.

 

Archivo: atm.py

class CajeroATM:

    def __init__(self, saldo_inicial: float):

        self.saldo = saldo_inicial

 

    def retirar(self, monto: float) -> str:

        if monto <= 0:

            return "Monto inválido"

        if monto > self.saldo:

            return "Fondos Insuficientes"

        self.saldo -= monto

        return "Retiro exitoso"

7. Cuestionario – Pregunta 3: Myers y las Pruebas de Error

 

"El objetivo de las pruebas es encontrar errores. Un buen caso de prueba es aquel que tiene alta probabilidad de detectar un error aún no descubierto."

— Glen Myers, The Art of Software Testing

 

¿Por qué el caso de fondos insuficientes es una prueba de error necesaria?

Glen Myers establece en su libro The Art of Software Testing que un software robusto no puede validarse únicamente con datos correctos. Las pruebas de error son igual de importantes porque representan situaciones reales que el usuario podría provocar.

 

En el caso del cajero ATM, el escenario donde se intenta retirar más dinero del disponible (saldo=100, retiro=150) es un caso de "entrada inválida esperada", que puede ocurrir en la realidad. Si el software no maneja explícitamente este caso:

 

•       Podría descontar dinero inexistente, generando saldo negativo.

•       Podría lanzar una excepción no controlada y crashear.

•       Podría comportarse de forma impredecible, afectando la confianza del usuario.

 

Por ello, Myers clasifica este tipo de prueba como necesaria: no prueba que el sistema hace lo correcto, sino que el sistema se defiende correctamente ante lo incorrecto. Esto es la base de la robustez del software.

 

En nuestro simulador, la prueba test_retiro_mayor_al_saldo_muestra_fondos_insuficientes cumple exactamente este rol: verifica que el cajero bloquea la transacción y notifica al usuario en lugar de procesarla incorrectamente.

 

 

8. Conclusiones

•       El flujo BDD → TDD permite definir el comportamiento esperado antes de programar, reduciendo errores de interpretación.

•       El archivo .feature actúa como documentación viva que cualquier persona puede entender.

•       Las 8 pruebas implementadas cubren clases de equivalencia válidas, inválidas y valores límite.

•       Siguiendo a Myers, probar casos de error es tan importante como probar casos exitosos para garantizar la robustez del sistema.
